# Token Optimization

## The Headline Number

If a naive multi-agent baseline costs $1.00 per substantive request, Flug should land between $0.30 and $0.40. That's the headline metric for the README and the eval dashboard.

This isn't achieved through one brilliant trick. It's achieved by stacking independent levers, each contributing 5-30% savings, that compound.

## Frame the Problem Correctly

Token optimization is fundamentally a **scheduling and budgeting problem**, not a prompt-engineering problem. Most engineers approach this by trying to write tighter system prompts. That's a 5% win at best. The real wins come from treating the pipeline like a compiler optimizing a program: which stages run, in what order, on which model, with what context, and with how much output.

## Six Categories of Token Waste

Ranked by typical impact:

| Category | Typical waste | Example |
|---|---|---|
| Redundant context propagation | 30-40% | Passing full chat history to every agent |
| Wrong model tier | 20-30% | Using a frontier model for tasks a small one handles |
| Unnecessary stages | 15-25% | Running security audit on a typo fix |
| Verbose outputs | 10-15% | Agents emitting essays when JSON would do |
| Cache misses | 5-15% | Re-computing identical sub-tasks |
| Over-large prompts | 5-10% | Loading the full knowledge base every call |

The first three together are 65-95% of waste. Focus there.

## The Token Optimizer

Build it as an explicit component, not a side effect of good prompting. The optimizer runs once at the start of each pipeline and outputs a budget allocation:

```python
class TokenBudget(BaseModel):
    total_input_budget: int        # ceiling for the whole run
    total_output_budget: int
    per_stage: dict[str, StageBudget]
    reserve: int                   # 15-20% held back for retries
    overflow_policy: Literal["abort", "downgrade", "trim"]

class StageBudget(BaseModel):
    input_max: int
    output_max: int                # passed to model as max_tokens
    model_tier: Literal["nano", "small", "medium", "large"]
    skippable: bool
```

Inputs: user request + complexity classification.
Outputs: budget. Every stage call goes through a wrapper that enforces it. If a stage exceeds its allocation, the wrapper trims context, downgrades the model, or aborts depending on policy.

This sounds bureaucratic but it's the only way to get predictable cost. Without explicit budgeting, token usage is whatever the agents felt like consuming, which is unbounded.

### Week 3 implementation order

The first implementation slice is not a new classifier or cache. It is accounting. The current provider adapters already return token and cost usage, but the harness must persist that usage on `StageRecord`, SQLite stage rows, run totals, and trajectory state before higher-level optimizer behavior can be trusted.

Week 3 therefore lands in this order:

1. Record provider usage at stage boundaries.
2. Enforce `StageBudget.output_max` and typed budget failures.
3. Add exact input-token preflight counting where providers support it (Anthropic first).
4. Add typed complexity classification and tier-specific planner shapes.
5. Add bounded summaries, prompt caching, confidence escalation, cost reporting, semantic cache, evals, and additional providers.

This ordering keeps every later optimization measurable. A cache hit, retry, escalation, or skipped stage only matters if the run ledger can show the token and cost effect.

## The Levers, Ranked by Impact

### 1. Hierarchical Summarization (biggest single win)

Every agent output exists in two forms:

- **Full output**: stored in SQLite, used for audit and replay
- **Summary**: 50-150 tokens, generated by a tiny call, used as downstream context

The orchestrator never forwards full outputs to later agents. It forwards summaries. By stage 8, accumulated context is ~400 tokens of summaries instead of ~8,000 tokens of raw outputs.

```python
async def stage_complete(stage_name: str, output: BaseModel):
    full_json = output.model_dump_json()
    summary = await summarize_for_downstream(full_json, stage_name)
    db.insert_stage(stage_name, full_json, summary)
    return summary  # this is what flows forward
```

The summarization itself costs tokens, but it's a nano-tier call, and savings downstream are 10-20x the cost. Net win is enormous.

### 2. Prompt Caching (the underused superpower)

Anthropic, OpenAI, and Google all support server-side prompt caching. Mark stable prefixes (system prompts, knowledge base content, project context.md) as cacheable. First call pays full price. Subsequent calls within TTL pay 10% (Anthropic) or 50% (OpenAI) of input cost for the cached portion.

For Flug this is huge. The bundled knowledge base, project context, and agent system prompts are all cacheable. Across an 8-stage run you cache once and pay 10% on the next 7 calls. This alone cuts input costs by 50-60%.

Anthropic uses 5-minute TTLs and a 1024-token minimum to be cacheable. Structure prompts so cacheable content is the prefix:

```python
messages = [
    {"role": "system", "content": [
        {"type": "text", "text": KNOWLEDGE_BASE, "cache_control": {"type": "ephemeral"}},
        {"type": "text", "text": PROJECT_CONTEXT, "cache_control": {"type": "ephemeral"}},
        {"type": "text", "text": agent_specific_prompt},  # not cached, varies
    ]},
    {"role": "user", "content": stage_input},
]
```

This is mandatory infrastructure, not an optimization. Build it in from day one.

### 3. Routing by Request Complexity

Not every request goes through the full pipeline. Orchestrator's first job: classify and pick a pipeline shape.

| Tier | Trigger | Pipeline | Token cost |
|---|---|---|---|
| Trivial | < 20 line edit, no logic change | Writer only | ~$0.01 |
| Light | Bug fix, single function | Writer + Static + Test | ~$0.05 |
| Standard | New feature, multiple files | Full execution pipeline | ~$0.20 |
| Substantial | New component or service | Full + design phase | ~$0.50 |
| Architectural | New service, integration | Full + extra critic passes | ~$1.00 |

Classifier is a 200-token nano call. Costs nothing. Routes ~60% of real requests away from the expensive pipeline. Single biggest cost lever in production.

### 4. Model Tiering with Confidence Escalation

Don't default to the strong model. Default to the cheap one. Each agent outputs a `confidence` field. Below threshold (typically 0.7), the orchestrator re-runs that specific stage on a stronger model.

In practice ~70% of stages succeed on small/nano. The ~30% that escalate to medium are the ones that genuinely needed it. You pay frontier prices only when warranted: roughly 3-4x cheaper than running a frontier model everywhere with no quality loss in evals.

Threshold isn't fixed. Tune per agent based on eval harness results. Static analyzer can tolerate 0.6 because tools catch most errors. Architect needs 0.85 because mistakes propagate.

### 5. Deterministic-First Architecture

Anything that can be answered with code, not LLMs, must be answered with code:

- AST parsing for code structure questions: free
- ripgrep for "where is X used": free
- pylint, mypy, ESLint, tsc for syntax and types: free
- Bandit, Semgrep for known vulnerability patterns: free
- pytest, jest for actual correctness: free
- Git diff for "what changed": free

The LLM is the last resort, not the first reach. ~40% of operations that look like LLM tasks are actually deterministic. Eliminating those calls is pure profit.

### 6. Diff-Based Iteration

When an agent fails and the writer revises, do not re-send the full codebase. Send the diff. The static analyzer doesn't re-analyze unchanged code. The test runner only re-runs affected tests.

For a 1,000-line file with a 30-line fix, this is a 97% reduction in code tokens for the iteration cycle. Iterations happen often (test failures, security flags, lint issues) so this matters cumulatively.

### 7. Just-In-Time Context Loading

The architect doesn't need the full codebase. It needs a directory tree and the ability to request specific files. Give agents tools, not pre-loaded context:

```python
class CodebaseTools:
    async def list_files(self, path: str) -> list[str]: ...
    async def read_file(self, path: str) -> str: ...
    async def search(self, query: str) -> list[Match]: ...
    async def find_symbol(self, name: str) -> list[Location]: ...
```

The agent decides what it needs. Most of the time it needs 2-3 files. Sometimes 10. You only pay for what's actually used. Same pattern Claude Code uses, and why it can navigate 100k-line codebases without exploding tokens.

### 8. Output Length Constraints

Tell agents their output budget directly in the prompt: *"Your output must be under 400 tokens."* Models follow this. Combined with `max_tokens` as a hard cap on the API call, you prevent runaway outputs.

Most agents don't need verbose outputs. The static analyzer's job is to emit a structured list of issues, not a 2,000-token essay. Cap it.

### 9. Semantic Caching Across Runs

Cache key: `sha256(canonical_input_schema + model + agent_version)`. Storage: SQLite. TTL: 24 hours.

Cache hit rates by stage (covered in `02-storage-and-memory.md`). Aggregate savings: 15-25% of total tokens once warm.

## The Optimizer Algorithm

Putting it together as actual control flow:

```python
async def run_pipeline(request: str) -> PipelineResult:
    # 1. Classify complexity (1 nano call, ~200 tokens)
    classification = await classifier.classify(request)

    # 2. Build budget based on tier
    budget = TokenBudget.for_tier(classification.tier)

    # 3. Build pipeline shape
    plan = pipeline_planner.plan(classification, budget)

    # 4. Pre-cache knowledge base if pipeline is substantial
    if plan.uses_knowledge_base:
        await prompt_cache.warm(plan.required_kb_files)

    # 5. Execute stages with budget enforcement
    summaries: dict[str, str] = {}
    for stage in plan.stages:
        # Check cache
        cached = await semantic_cache.get(stage.input_hash)
        if cached:
            summaries[stage.name] = cached.summary
            continue

        # Allocate this stage's slice of budget
        stage_budget = budget.per_stage[stage.name]

        # Run on cheap model first
        try:
            output = await run_with_budget(stage, stage_budget, model="nano")
        except LowConfidence:
            output = await run_with_budget(stage, stage_budget, model="medium")

        # Summarize for downstream
        summary = await summarize(output, target_tokens=100)
        summaries[stage.name] = summary

        # Persist full output
        await db.save_stage(stage.name, output)
        await semantic_cache.set(stage.input_hash, output, summary)

        # Update remaining budget
        budget.consume(output.tokens_used)

        # Trigger overflow policy if needed
        if budget.overspent():
            plan = plan.downgrade_remaining()

    return PipelineResult(...)
```

This is approximate but the structure is real. The optimizer is just disciplined budget tracking plus the levers above, applied consistently.

## Compounding the Levers

| Lever | Savings (as % of remainder) |
|---|---|
| Naive baseline (frontier model for everything, full context every stage) | 100% |
| Hierarchical summarization | -35% |
| Model tiering (small default, escalate as needed) | -25% |
| Prompt caching on knowledge base | -20% |
| Deterministic-first (tools before LLMs) | -15% |
| Diff-based iteration | -10% |
| Semantic cache | -15% |
| **Final cost** | ~33% of naive baseline |

Each lever is independently easy. Built well, the stack lands at the 30-40% target. Built poorly, you regress to 70-80%. The difference is engineering discipline, not raw cleverness.

## Eval Metrics

The eval harness tracks token efficiency, not just correctness. Dashboard shows:

- Cost per run (USD)
- Tokens per run (input, output, cached, separately)
- Cost per successful test pass
- Cost ratio vs naive baseline (the headline metric)
- Cache hit rate by stage
- Model tier distribution (% of calls per tier)
- Confidence escalation rate (% of stages that needed upgrade)
- Stage skip rate (% of stages skipped by orchestrator)

The escalation rate is the tuning knob. If over 50%, prompts at the cheap tier are too weak. If under 10%, you're paying too much because the cheap tier could have handled even more cases.

## Known Gaps

### `fallback_on` and `fallback` config fields (not wired)

`StageSettings.fallback_on` and `StageSettings.fallback` in `flug/config.py` are configuration
placeholders. They define when and where to fall back (e.g., on rate limits or timeouts to
alternative providers), but the runner and registry do not read or act on them. Provider
fallback switching is not yet implemented. These fields are reserved for future use.

### Cross-run memory (Track M, post-v0.1)

The semantic cache covers within-run deduplication. Cross-run learning (persisting
discovered test commands, project conventions, known patterns so the pipeline does not
re-discover them on every run) is the memory sidecar's job (Track F/M3-M7), deferred to
post-v0.1.
